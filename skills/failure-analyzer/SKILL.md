# taichi Failure Analyzer Skill

## 1. Overview

This Skill lets an AI Agent read the structured failure context (`FailureContext`) generated by taichi after a test failure, and analyze the failure root cause by combining service logs with project source code, producing an actionable root cause analysis report.

This Skill is the "root cause analysis" stage of the taichi ↔ AI Agent bidirectional integration loop. It consumes the failure context produced by `taichi-test-runner` and provides the fix basis for `taichi-code-fixer`.

## 2. When to Invoke This Skill

**Mandatory invocation scenarios**:
- `taichi-test-runner` run result has `summary.failed > 0`
- The user mentions "analyze failure", "why did the test fail", "locate the bug", "root cause analysis"
- A copilot round has test failures and needs to provide a basis for fix decisions
- The failure context file has been generated in the `reports/` directory

**Do not invoke scenarios**:
- All tests passed (no analysis needed)
- Only need to re-run tests (use `taichi-test-runner` / `taichi-regression-runner`)
- Only need to apply a fix patch (use `taichi-code-fixer`)

## 3. Input Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `failure_context_path` | string | Yes | Failure context JSON file path, generated by taichi in the `reports/` directory after a test failure, named like `failures-round-1-<timestamp>.json` |

## 4. Failure Context Format (FailureContext JSON)

When a test has failed cases, taichi writes the structured failure context to the `reports/` directory as the information exchange contract between taichi and the AI Agent. The full structure is as follows:

```json
{
  "project_name": "tickraft",
  "base_url": "http://127.0.0.1:8080",
  "timestamp": "2026-07-17T10:30:00Z",
  "project_root": "/Users/derekray/workspace/auzekalabs/tickraft",
  "env_log_path": "/tmp/taichi-tickraft-env.log",
  "reports_dir": "/Users/derekray/workspace/auzekalabs/tickraft/taichi/reports",
  "total_cases": 24,
  "passed_cases": 22,
  "failed_cases": [
    {
      "skill_name": "ui",
      "name": "ui:home_page_render",
      "message": "expected status 200, got 500",
      "error": "GET http://127.0.0.1:8080/ returned 500",
      "duration": "1.23s"
    }
  ]
}
```

### Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `project_name` | string | Project under test name |
| `base_url` | string | Base URL of the service under test (empty when no environment is started) |
| `timestamp` | string | Failure context generation time (RFC3339, UTC) |
| `project_root` | string | Absolute path of the project under test root directory, used to locate source code |
| `env_log_path` | string | Environment (service) log file path, used to check server-side errors |
| `reports_dir` | string | Report output directory, containing the JSON / JUnit XML / summary of this run |
| `total_cases` | int | Total number of cases in this run |
| `passed_cases` | int | Number of passed cases |
| `failed_cases[]` | array | List of failed case details |
| `failed_cases[].skill_name` | string | Name of the skill that produced the case (e.g. `api`, `ui`) |
| `failed_cases[].name` | string | Case identifier |
| `failed_cases[].message` | string | Human-readable description of the failure |
| `failed_cases[].error` | string | String form of the underlying error (if any) |
| `failed_cases[].duration` | string | Case execution duration |

> Contract definition source: [`pkg/failure/failure.go`](../../pkg/failure/failure.go)

## 5. Analysis Steps

After receiving the failure context, execute in the following order:

### Step 1: Read the Failure Context

```
Read failure_context_path → parse FailureContext JSON
                          → extract project_name / project_root / env_log_path
                          → iterate failed_cases to get each failed case's name / message / error
```

### Step 2: Inspect Service Logs

```
Read the service log pointed to by env_log_path
  → search for ERROR / panic / stack trace within the failure time window
  → correlate with error info in failed_cases
  → identify server-side exceptions (nil dereference, connection refused, timeout, missing config, etc.)
```

If `env_log_path` is empty (no environment started), skip this step and note it in the report.

### Step 3: Locate Relevant Source Code

```
Using project_root as the root
  → locate handler / business code based on module/route clues in failed_cases[].name
  → locate the exact position based on file:line in stack traces
  → read relevant source code snippets to understand the control flow
```

### Step 4: Analyze Root Cause

```
Combine case failure info + service logs + source code context
  → determine the root cause category (nil dereference / logic error / missing config / dependency unavailable / assertion mismatch / environment issue)
  → assess the impact scope (whether other cases are affected)
  → provide a suggested fix (specific to file and line number)
```

## 6. Output Format (Root Cause Analysis Report)

After analysis, output a structured root cause analysis report. This report serves as the input basis for `taichi-code-fixer`:

```json
{
  "project_name": "tickraft",
  "analyzed_at": "2026-07-17T10:31:00Z",
  "total_failures": 2,
  "root_causes": [
    {
      "case_name": "ui:home_page_render",
      "skill_name": "ui",
      "root_cause": "Homepage handler panics due to nil dereference caused by missing config",
      "category": "nil_dereference",
      "evidence": [
        "env_log: 2026-07-17T10:29:55Z panic: runtime error: invalid memory address",
        "source: internal/handler/home.go:42 accesses config.Theme without nil check"
      ],
      "affected_files": [
        "internal/handler/home.go"
      ],
      "suggested_fix": "Add a nil check at home.go:42; fall back to default theme when config is missing",
      "confidence": "high"
    }
  ],
  "summary": "Both failed cases stem from a nil dereference in the homepage handler; fixing home.go resolves both."
}
```

### Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `project_name` | string | Project under test name |
| `analyzed_at` | string | Analysis completion time (RFC3339) |
| `total_failures` | int | Total number of failed cases |
| `root_causes[]` | array | List of root causes, one per failed case |
| `root_causes[].case_name` | string | Failed case name |
| `root_causes[].skill_name` | string | Skill name |
| `root_causes[].root_cause` | string | One-sentence description of the root cause |
| `root_causes[].category` | string | Root cause category (`nil_dereference` / `logic_error` / `config_missing` / `dependency_unavailable` / `assertion_mismatch` / `env_issue` / `unknown`) |
| `root_causes[].evidence[]` | string[] | Evidence chain (log snippets, source code locations) |
| `root_causes[].affected_files[]` | string[] | Affected source files (relative to project_root) |
| `root_causes[].suggested_fix` | string | Suggested fix |
| `root_causes[].confidence` | string | Confidence level (`high` / `medium` / `low`) |
| `summary` | string | Overall conclusion |

## 7. Integration with Other Skills

- **Upstream**: `taichi-test-runner` produces the failure context file
- **Downstream**: `taichi-code-fixer` generates fixes based on this report's `affected_files` and `suggested_fix`

## 8. Output Self-check List

- [ ] The failure context JSON was successfully parsed
- [ ] The service log pointed to by `env_log_path` was inspected
- [ ] Source code was located using `project_root` as the baseline (avoiding path errors)
- [ ] Each failed case has a root cause and affected files
- [ ] The root cause category is accurately classified
- [ ] The suggested fix is specific to file and line number
